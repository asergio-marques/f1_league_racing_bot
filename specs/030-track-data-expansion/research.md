## 2. Track Name Mapping (Old Short Names → New Canonical Circuit Names)

**Question**: The existing `TRACK_IDS` dict uses 27 short names (e.g. "Australia", "Japan").
The spec's seed table uses full circuit names (e.g. "Albert Park Circuit",
"Suzuka International Racing Course"). How do existing `rounds.track_name` values
interoperate with the new schema?

**Decision**: The migration includes `UPDATE rounds SET track_name = '<new>' WHERE track_name = '<old>'`
for all 27 existing circuits. Phase 1 weather generation resolves `(mu, sigma)` via
`SELECT mu, sigma FROM tracks WHERE name = ?` — so `rounds.track_name` must use the
same canonical name that `tracks.name` stores.

**Rationale**: Simplest path to correctness. No need for a dual-name lookup or legacy-name
column. Existing rounds (if any) become consistent with the new schema in one migration step.

**Alternative considered**: Add a `legacy_name TEXT` column to `tracks` and use `WHERE name = ? OR legacy_name = ?`
in Phase 1. Rejected: adds query complexity and a permanent schema wart for a one-time migration concern.

## 5. `track_service.py` Architecture Post-Migration

**Question**: Should `track_service.py` be deleted entirely or repurposed?

**Decision**: Repurpose `track_service.py`. Remove all `track_rpc_params` CRUD functions and
replace with two DB-backed query functions for the new `tracks` table:
- `get_all_tracks(db) -> list[dict]` — used by `/track list` and autocomplete
- `get_track_by_name(db, name: str) -> dict | None` — used by Phase 1 and round validation

**Rationale**: Preserves module system convention (services own DB interaction). The cog and
phase1 service import from `track_service`; keeping the file avoids import refactoring in those
callers. Renaming functions is a smaller change than wiring inline queries.

**Alternative considered**: Delete `track_service.py` and inline queries directly in the cog
and `phase1_service.py`. Rejected: breaks existing import pattern, harder to test.

**Evidence**: `track_rpc_params` was created in migration `005_track_rpc_params.sql`.
Any installation that ran at least migration 005 will have it. The `IF EXISTS` guard handles
clean installations that never ran any prior migration file.

## 7. Autocomplete Feed

**Question**: After `TRACK_IDS` is removed, how does autocomplete for `/round add`,
`/round amend`, and `/track list` get its data?

**Decision**: Autocomplete handlers query `SELECT id, name FROM tracks ORDER BY id` via
`track_service.get_all_tracks(db)`. The label format is `"{id:02d} – {name}"` (matching the
current zero-padded-ID format but now using the canonical circuit name).

**Rationale**: DB-backed autocomplete is consistent with the new architecture. The 28-row
query is trivially fast. The zero-padded label format preserves the user experience for
admins who have memorised numeric IDs.

**Note**: The `season_cog.py` round commands currently access `bot.db_path` via
`get_connection`; autocomplete handlers are `async def` and can use the same pattern.
