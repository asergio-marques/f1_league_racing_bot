## New Tables

### `tracks`

Central registry for every F1 circuit. Authoritative source for weather parameter
resolution after the `track_rpc_params` table is dropped.

| Column       | Type    | Constraints          | Notes                                        |
|--------------|---------|----------------------|----------------------------------------------|
| `id`         | INTEGER | PK, NOT NULL         | 1-indexed stable key; matches spec seed data |
| `name`       | TEXT    | NOT NULL, UNIQUE     | Canonical circuit name (e.g. "Albert Park Circuit") |
| `gp_name`    | TEXT    | NOT NULL             | Grand prix event name (e.g. "Australian Grand Prix") |
| `location`   | TEXT    | NOT NULL             | City/venue string                            |
| `country`    | TEXT    | NOT NULL             | Country string                               |
| `mu`         | REAL    | NOT NULL             | Mean rain probability (Beta distribution)    |
| `sigma`      | REAL    | NOT NULL             | Dispersion (Beta distribution)               |

**Relationships**: No outbound FK constraints. `rounds.track_name` (TEXT) stores the
canonical circuit name matching `tracks.name`; the relationship is by value, not by FK.
`track_records.track_id` and `lap_records.track_id` are FK → `tracks.id`.

**Seed**: 28 rows inserted via `INSERT OR IGNORE` at migration time (idempotent).

**Mutability**: Rows are inserted by migration only. No runtime insert/update/delete
commands are exposed in this increment.

### `track_records`

All-time best performance at a circuit for a given tier and session type.

| Column         | Type    | Constraints              | Notes                                              |
|----------------|---------|--------------------------|----------------------------------------------------|
| `id`           | INTEGER | PK AUTOINCREMENT         |                                                    |
| `track_id`     | INTEGER | NOT NULL, FK → tracks.id |                                                    |
| `tier`         | INTEGER | NOT NULL                 | Division tier level                                |
| `session_type` | TEXT    | NOT NULL                 | One of: SHORT_QUALIFYING, SHORT_SPRINT_QUALIFYING, LONG_SPRINT_RACE, SHORT_FEATURE_QUALIFYING, LONG_FEATURE_RACE |
| `game`         | TEXT    | NOT NULL                 | Free-form game identifier, e.g. "EA F1 25"        |
| `season_number`| INTEGER | NOT NULL                 |                                                    |
| `round_number` | INTEGER | NOT NULL                 |                                                    |
| `lap_time`     | TEXT    | NOT NULL                 | Normalised lap-time string, e.g. "1:23.456"       |
| `driver_id`    | INTEGER | NOT NULL                 | Discord user ID of the driver who set the record   |

**Unique constraint**: `(track_id, tier, session_type)` — one record per circuit × tier × session.

**Mutability**: Populated by a future increment; no insert/update commands in this feature.

**Validation**: No constraint is added at the DB level restricting `session_type`; enforcement
is at the data layer when writes are added in a future increment.

### `lap_records`

All-time fastest race lap at a circuit for a given tier and race session type.
Qualifying sessions are excluded.

| Column         | Type    | Constraints              | Notes                                              |
|----------------|---------|--------------------------|----------------------------------------------------|
| `id`           | INTEGER | PK AUTOINCREMENT         |                                                    |
| `track_id`     | INTEGER | NOT NULL, FK → tracks.id |                                                    |
| `tier`         | INTEGER | NOT NULL                 | Division tier level                                |
| `session_type` | TEXT    | NOT NULL                 | One of: LONG_SPRINT_RACE, LONG_FEATURE_RACE only   |
| `game`         | TEXT    | NOT NULL                 | Free-form game identifier, e.g. "EA F1 25"        |
| `season_number`| INTEGER | NOT NULL                 |                                                    |
| `round_number` | INTEGER | NOT NULL                 |                                                    |
| `lap_time`     | TEXT    | NOT NULL                 | Normalised lap-time string, e.g. "1:23.456"       |
| `driver_id`    | INTEGER | NOT NULL                 | Discord user ID of the driver who set the record   |

**Unique constraint**: `(track_id, tier, session_type)` — one lap record per circuit × tier × race session.

**Mutability**: Populated by a future increment; no insert/update commands in this feature.

## Dropped Tables

### `track_rpc_params` (DROPPED)

Previously stored per-server weather parameter overrides (`mu_rain_pct`, `sigma_rain_pct`,
`track_name`, `server_id`). Dropped cleanly by the migration. All overrides are permanently
discarded; the `tracks` table values are the authoritative source post-migration.

---

## Modified Data

### `rounds.track_name` (UPDATED in migration)

`rounds.track_name` remains a TEXT column (no FK added). The migration includes UPDATE
statements to rename the 27 old short names to the new canonical circuit names, ensuring
Phase 1 weather generation resolves correctly via `WHERE tracks.name = rounds.track_name`.

Full mapping: see `research.md` § 2.

## Entities (Application Layer)

### `Track` (new)

```python
@dataclass
class Track:
    id: int
    name: str           # canonical circuit name
    gp_name: str        # grand prix event name
    location: str
    country: str
    mu: float
    sigma: float
```

Returned by `track_service.get_all_tracks(db)` and `track_service.get_track_by_name(db, name)`.

### `TrackRecord` / `LapRecord` (structural, no application class yet)

These tables are created at migration time as structural prerequisites. No dataclass or service
method is defined for them in this increment; they are populated by a future feature.

---

## State Transitions

No new state machines are introduced. Phase 1 weather generation remains unchanged in
structure; only the data source for `(mu, sigma)` changes (DB row instead of Python dict).

The `/division amend` command mutates a `divisions` row (tier, name, or mention_role_id)
during SETUP state only. No status transition is triggered.

---

## Indices

The migration adds no explicit indices beyond the primary keys and the UNIQUE constraints.
The `tracks` table will see at most 28 rows; a linear scan on every lookup is negligible.
The `track_records` and `lap_records` tables start empty; indices can be added when records
management is implemented.
